## 진행 방식

1. `git log main..HEAD`와 `git diff main...HEAD --stat`으로 브랜치의 전체 변경사항을 파악한다
2. 커밋되지 않은 변경사항이 있으면 먼저 `/commit`을 제안한다
3. remote에 push되지 않은 커밋이 있으면 `git push -u origin <branch>`로 push한다
4. PR 제목과 본문을 작성하고 사용자에게 확인받는다
5. `gh pr create`로 PR을 생성한다

## PR 제목 규칙

```
<type>: <요약> (한국어)
```

- commit.md의 type/규칙과 동일
- 여러 커밋이면 전체를 아우르는 요약으로 작성
- 70자 이내

## PR 본문 형식

리뷰어(메인 담당자)가 변경 맥락을 빠르게 파악할 수 있도록 작성한다.
"무엇을 왜 바꿨는지"를 중심으로, 코드를 읽지 않아도 변경 의도를 이해할 수 있어야 한다.

```markdown
## 배경
왜 이 변경이 필요한가. 어떤 문제를 해결하는가.
(1~2문장으로 맥락 설명)

## 변경 내용
- 파일/영역별로 **무엇을 어떻게** 바꿨는지 bullet으로 정리
- 단순 파일 목록이 아니라, 변경의 의도와 영향을 설명
- 삭제한 것이 있으면 왜 삭제했는지도 명시

## 리뷰 포인트
리뷰어가 특히 봐줬으면 하는 부분이나, 논의가 필요한 결정 사항.
없으면 생략.

## 검증
- [ ] 테스트/검증 방법 체크리스트

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

## 주의

- `$ARGUMENTS`가 있으면 PR 제목/본문 힌트로 활용한다
- base 브랜치는 `main`을 기본으로 한다
- draft PR이 필요하면 사용자에게 확인 후 `--draft` 플래그를 추가한다
- PR 생성 후 URL을 출력한다
