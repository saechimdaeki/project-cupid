## 진행 방식

1. `git diff`와 `git status`로 변경사항을 파악한다
2. `$ARGUMENTS`가 있으면 커밋 메시지 힌트로 활용한다
3. 현재 브랜치가 `main`이면 새 브랜치명을 제안하고 사용자에게 확인받는다
   - 브랜치명 형식: `<type>/<간단한-설명>` (예: `feat/candidate-filter`)
   - main이 아니면 현재 브랜치에서 그대로 진행한다
4. 커밋 메시지를 작성하고 사용자에게 확인받는다
5. 브랜치 생성(필요 시) → 커밋 순으로 실행한다

## 커밋 메시지 규칙

```
<type>: <요약> (한국어)
```

**type**
- `feat` — 새 기능
- `fix` — 버그 수정
- `chore` — 설정, 의존성, 기타
- `refactor` — 동작 변화 없는 코드 변경
- `style` — UI/스타일 변경

**규칙**
- 요약은 한국어로, 명사형으로 끝낸다 (예: "후보 필터링 기능 추가")
- 50자 이내
- 변경 이유가 불명확하면 본문 추가

## 주의

- `.env` 등 민감한 파일이 스테이징에 포함되어 있으면 커밋 전에 반드시 경고한다
- 관련 없는 변경사항이 섞여 있으면 분리 커밋을 제안한다
